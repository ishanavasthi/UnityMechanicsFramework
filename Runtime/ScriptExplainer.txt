PURPOSE:
A God-of-War-style throw-and-recall weapon system for Unity. The weapon
can be thrown forward, embed into surfaces (including moving platforms),
and recalled back to the player's hand along a curved Bezier path.
Demonstrates state-based physics management, parenting strategies,
IPhysicsAdapter usage, EventBus decoupling, and ScriptableObject config.

DEPENDENCIES:
- GameplayMechanicsUMFOSS.Core (StateMachine, EventBus, IState)
- GameplayMechanicsUMFOSS.Physics (IPhysicsAdapter, Physics3DAdapter)
- GameplayMechanicsUMFOSS.Utils (BezierUtils, LayerMaskUtils, PhysicsUtils, CameraShake)
- TextMeshPro (demo UI only)

SCRIPT: BoomerangWeapon_UMFOSS.cs
FILE: Runtime/Combat/BoomerangWeapon/BoomerangWeapon_UMFOSS.cs

Code    : public class BoomerangWeapon_UMFOSS : MonoBehaviour
Explain : The orchestrator MonoBehaviour. It owns the state machine and
          is the only script attached to the weapon GameObject. All logic
          is delegated to individual state classes — this class only
          wires dependencies and forwards Unity lifecycle events.

Code    : [SerializeField] private BoomerangConfig config;
Explain : ScriptableObject holding every tunable value (throw force,
          damage, return speed, layers, etc.). Using a SO means you can
          create multiple weapon variants (axe, spear, hammer) by
          swapping the asset — no code duplication needed.

Code    : [SerializeField] private Transform handSocket;
Explain : An empty child Transform on the player's camera rig. When the
          weapon is "equipped" it parents itself here, so it moves with
          the player's hand. When thrown, it un-parents to become a
          free-flying physics object.

Code    : [SerializeField] private Transform visualPivot;
Explain : A child Transform that holds all mesh renderers. Spin rotation
          is applied to this pivot, NOT to the root. This keeps the
          root's forward vector clean for movement direction calculations.

Code    : var physics = GetComponent<IPhysicsAdapter>();
Explain : Instead of referencing Rigidbody directly, we use the
          framework's IPhysicsAdapter interface. This makes the weapon
          compatible with both Physics3DAdapter and future Physics2DAdapter
          without changing any weapon code.

Code    : stateMachine = new StateMachine<WeaponStateKey>();
Explain : Creates the generic state machine keyed by WeaponStateKey enum.
          Four states are registered: Equipped, Thrown, Embedded, Recalling.
          The machine handles Enter/Update/Exit lifecycle forwarding.

Code    : private void OnCollisionEnter(Collision collision)
Explain : Unity's physics callback. Only processed when in Thrown state —
          other states are kinematic so collisions won't fire. Delegates
          to ThrownState.HandleCollision() which checks the stuck layer
          before transitioning.

Code    : public void Throw(Vector3 direction)
Explain : Public API called by input controllers. Guards against throwing
          when not equipped. Sets throw direction and origin on the context
          object, then transitions the state machine.

Code    : public void Recall()
Explain : Public API for recall. Guards against recalling when already
          equipped or already recalling. Works from both Thrown and
          Embedded states.

SCRIPT: BoomerangConfig.cs
FILE: Runtime/Combat/BoomerangWeapon/BoomerangConfig.cs
====================================================

Code    : [CreateAssetMenu(fileName = "NewBoomerangConfig", menuName = "UMF/Combat/Boomerang Config")]
Explain : Registers this ScriptableObject in Unity's Create menu so
          designers can create weapon variants without touching code.

Code    : [SerializeField] private AnimationCurve throwCurve
Explain : An animation curve that shapes the weapon's speed over its
          flight distance. X axis = 0-1 normalized distance, Y = speed
          multiplier. A flat curve gives constant speed; a declining
          curve makes the weapon slow down before auto-recall.

Code    : [SerializeField] private float arcHeightRatio = 0.3f;
Explain : Controls the upward bulge of the Bezier return path. A value
          of 0.3 means the control point sits 30% of the return distance
          above the midpoint, creating a visible arc. Set to 0 for a
          straight-line return.

Code    : [SerializeField] private LayerMask hurtboxLayer;
Explain : Physics layers the weapon checks for overlap during flight.
          Targets on this layer with an IDamageable component take damage.

Code    : [SerializeField] private LayerMask stuckLayer;
Explain : Physics layers where the weapon embeds on collision. Walls,
          floors, platforms — anything on this layer catches the weapon.

Code    : public class BoomerangContext
Explain : A plain C# class (not MonoBehaviour) that holds shared
          references for all states. This is the "composition root"
          pattern — instead of states reaching back into the MonoBehaviour,
          they receive everything they need through this context.
          Benefits: testable without Unity lifecycle, no circular refs.

Code    : public IPhysicsAdapter Physics;
Explain : The physics adapter reference shared by all states. States call
          Physics.SetKinematic(), Physics.AddForce(), and Physics.Velocity
          through this interface rather than holding a raw Rigidbody.

Code    : public Collision LastCollision;
Explain : Stores collision data from ThrownState so EmbeddedState.Enter()
          can access the surface and contact point. This avoids passing
          collision data through the state machine transition.

SCRIPT: BoomerangEvents.cs

====================================================

Code    : public struct WeaponThrownEvent
Explain : Published in ThrownState.Enter(). Carries the throw direction
          and weapon reference. Audio systems subscribe to play a whoosh;
          UI can show a "weapon in flight" indicator.

Code    : public struct WeaponStuckEvent
Explain : Published in EmbeddedState.Enter(). Carries the surface
          GameObject, contact point, and weapon ref. VFX systems spawn
          impact sparks at the contact point; audio plays an embed sound.

Code    : public struct WeaponRecallStartedEvent
Explain : Published in RecallingState.Enter(). Signals that the recall
          wiggle has started and the weapon is about to arc back.

Code    : public struct WeaponCaughtEvent
Explain : Published in RecallingState.Update() when the weapon reaches
          the hand. Camera shake, catch sound, and controller rumble
          all trigger from this single event.

Code    : public struct WeaponHitEvent
Explain : Published by HitDetector for each new target hit during flight
          or recall. Carries the target, weapon, hit position, and damage
          dealt. Hit-stop, sparks, and damage numbers all subscribe here.

All events are STRUCTS, not classes. This avoids heap allocations on
every publish, keeping the EventBus GC-friendly during gameplay.

SCRIPT: EquippedState.cs
FILE: Runtime/Combat/BoomerangWeapon/States/EquippedState.cs
====================================================

Code    : ctx.Physics.SetKinematic(true);
Explain : When equipped, the weapon must not respond to physics forces.
          SetKinematic zeroes velocity and angular velocity BEFORE
          toggling isKinematic to prevent stored forces from causing
          an impulse on the next mode change.

Code    : ctx.Transform.SetParent(ctx.HandSocket);
Explain : PARENTING STRATEGY — The weapon becomes a child of the hand
          socket so it moves with the player's camera rig. This is why
          the hand socket is an empty Transform on the camera holder,
          not the player root. When the player looks around, the weapon
          follows naturally.

Code    : ctx.Transform.localPosition = Vector3.zero;
          ctx.Transform.localRotation = Quaternion.identity;
Explain : Snaps the weapon to exact hand position and neutral rotation.
          This gives the "snappy catch" feel — no lerp, no delay.
          The weapon is immediately ready to throw again.

SCRIPT: ThrownState.cs
FILE: Runtime/Combat/BoomerangWeapon/States/ThrownState.cs
====================================================

Code    : ctx.Transform.SetParent(null);
Explain : PARENTING STRATEGY — Detaching from the hand socket so the
          weapon becomes a root-level object in the scene hierarchy.
          This is essential: a parented object inherits its parent's
          transform, so physics would fight the parent's movement.

Code    : ctx.Physics.SetKinematic(false);
          ctx.Physics.AddForce(ctx.ThrowDirection * ctx.Config.ThrowForce, ForceMode.VelocityChange);
Explain : STATE-BASED PHYSICS — Switching from kinematic (equipped) to
          dynamic (thrown). VelocityChange ignores mass, giving
          consistent throw speed regardless of weapon weight. The
          IPhysicsAdapter abstracts the underlying Rigidbody.

Code    : float curveMultiplier = ctx.Config.ThrowCurve.Evaluate(normalizedDist);
          ctx.Physics.Velocity = ctx.ThrowDirection * (ctx.Config.ThrowForce * curveMultiplier);
Explain : Shapes velocity over distance using the designer's animation
          curve. This overrides physics velocity each frame to give
          precise control over the flight profile. A declining curve
          creates a natural "slowing down" before auto-recall.

Code    : if (distanceTravelled >= ctx.Config.MaxRange)
              ctx.StateMachine.ChangeState(WeaponStateKey.Recalling);
Explain : Auto-recall safety net. If the weapon hasn't hit anything by
          max range, it starts returning. Prevents weapons flying off
          into infinity.

Code    : if (!LayerMaskUtils.IsInMask(collision.gameObject.layer, ctx.Config.StuckLayer))
              return;
Explain : Only embed in surfaces on the StuckLayer. Hitting an enemy
          (HurtboxLayer) deals damage via HitDetector but does NOT
          embed — the weapon passes through. This is the God-of-War
          behavior: axe embeds in walls, not in enemies.

Code    : ctx.LastCollision = collision;
Explain : Stores the collision reference so EmbeddedState can access the
          surface transform and contact point in its Enter() method.

====================================================
SCRIPT: EmbeddedState.cs
FILE: Runtime/Combat/BoomerangWeapon/States/EmbeddedState.cs
====================================================

Code    : ctx.Physics.SetKinematic(true);
Explain : STATE-BASED PHYSICS — Transition from dynamic to kinematic.
          The weapon must stop responding to physics forces immediately
          when it embeds. SetKinematic clears all velocity first.

Code    : ctx.Transform.SetParent(collision.transform);
Explain : PARENTING STRATEGY — THIS IS WHY WE PARENT ON IMPACT.
          By parenting the weapon to the hit surface's transform, the
          weapon automatically moves with that surface. If the surface
          is a moving platform, elevator, or rotating door, the embedded
          weapon follows it perfectly without any Update() tracking code.
          This is far more reliable than storing a world position and
          manually updating it — the Unity transform hierarchy does all
          the work for free.

Code    : ContactPoint contact = collision.GetContact(0);
Explain : Gets the first contact point from the collision. This gives us
          the exact world-space position and surface normal where the
          weapon hit. Used to publish the WeaponStuckEvent so VFX can
          spawn sparks at the right location.

Code    : ctx.VisualHandler.ResetRotation();
Explain : Stops the spin immediately. The visual pivot's local rotation
          snaps to identity so the weapon appears lodged at a natural
          angle, not mid-spin.

SCRIPT: RecallingState.cs
FILE: Runtime/Combat/BoomerangWeapon/States/RecallingState.cs
====================================================

Code    : Vector3 worldPos = ctx.Transform.position;
          ctx.Transform.SetParent(null);
          ctx.Transform.position = worldPos;
Explain : PARENTING STRATEGY — Un-parent from the surface BEFORE starting
          the return. We cache world position first because SetParent(null)
          would otherwise move the weapon to its local position in world
          space (which could be wildly wrong if the surface has moved).

Code    : wiggleOrigin = returnStartPos;
          wiggleTimer = 0f;
          isWiggling = true;
Explain : Before the Bezier arc begins, the weapon "wiggles" in place for
          ~0.12 seconds. This telegraphs the recall to the player and
          gives the satisfying "weapon is pulling free" feel before it
          arcs back. The wiggle uses small sinusoidal offsets.

Code    : bezierControlPoint = BezierUtils.ComputeArcControlPoint(...)
Explain : VECTOR MATH — Computes the control point for a quadratic
          Bezier curve between the weapon and the hand socket. The
          control point is offset upward (arcHeightRatio) and sideways
          (arcSideRatio) from the midpoint, creating the signature
          curved return path instead of a boring straight line.

Code    : currentSpeed += ctx.Config.ReturnAcceleration * Time.deltaTime;
Explain : The return accelerates over time for the "snap" catch feel.
          Starts slow (weapon pulling free), ends fast (snapping into
          hand). This acceleration makes the catch feel responsive
          even from long distances.

Code    : progress += currentSpeed * Time.deltaTime / totalDistance;
Explain : Progress is normalized 0-1 along the Bezier curve. Dividing
          by total distance ensures consistent timing regardless of
          how far the weapon was thrown.

Code    : Vector3 targetPos = BezierUtils.QuadraticBezier(returnStartPos, bezierControlPoint, ctx.HandSocket.position, progress);
Explain : VECTOR MATH — Evaluates the Bezier curve at the current
          progress. The curve goes: weapon position → arc control point
          → hand socket. Using the hand socket's CURRENT position (not
          cached) means the endpoint updates if the player moves during
          recall.

Code    : ctx.Transform.forward = toHand.normalized;
Explain : Points the weapon toward the hand as it returns. This keeps
          the weapon visually oriented along its flight path.

Code    : if (distToHand <= ctx.Config.CatchDistance || progress >= 1f)
Explain : Two catch conditions: distance-based (catches early if player
          walks toward weapon) and progress-based (safety net to prevent
          infinite loops if floating point drift prevents exact arrival).

SCRIPT: HitDetector.cs
FILE: Runtime/Combat/BoomerangWeapon/HitDetector.cs
====================================================

Code    : private readonly HashSet<Collider> hitTargets = new HashSet<Collider>();
Explain : Tracks which colliders have already been hit this flight.
          Without this, OverlapSphere would hit the same enemy every
          frame, dealing thousands of damage per second. The HashSet
          ensures each target is hit exactly once per flight phase.

Code    : public void ClearHitTargets(bool returnFlight = false)
Explain : Called when a new flight phase starts (throw or recall). Clears
          the hit set so targets can be hit again on the return trip.
          The returnFlight flag switches damage calculation between
          throwDamage and returnDamage.

Code    : float radius = weaponCollider != null ? weaponCollider.bounds.extents.magnitude : 0.5f;
Explain : Uses the weapon's collider bounds to determine overlap radius.
          Falls back to 0.5f if no collider is assigned, making the
          system resilient to missing references.

Code    : var damageable = hits[i].GetComponentInParent<IDamageable>();
Explain : Uses GetComponentInParent to find the IDamageable interface on
          the hit object or any of its parents. This supports hurtbox
          child colliders that are separate from the main health object.

Code    : float actualDamage = Mathf.Min(damageData.Amount, currentHealth);
Explain : Prevents overkill — damage is clamped to remaining health.
          This matters for damage numbers: showing "25" when the enemy
          only had 3 HP left would feel wrong.

Code    : ApplyKnockback(damageData);
Explain : Adds an impulse force in the knockback direction. Uses
          ForceMode.Impulse for immediate, frame-independent pushback.
          Direction is normalized so knockback force is consistent
          regardless of weapon travel direction magnitude.

Code    : EventBus.Publish(new DamageDealtEvent { ... });
Explain : EVENT DRIVEN — Published on every damage instance. Any system
          can subscribe: floating damage numbers, score tracking,
          analytics, sound effects. None need a reference to Damageable.

Code    : EventBus.Publish(new TargetKilledEvent { ... });
Explain : Published only when health reaches zero. Kill feeds, XP grants,
          achievement tracking, and death animations all subscribe to
          this single event.

Code    : private void SetRendererColor(Color color)
          propertyBlock.SetColor(BaseColorId, color);
          hitFlashRenderer.SetPropertyBlock(propertyBlock);
Explain : Uses MaterialPropertyBlock instead of modifying material.color.
          MaterialPropertyBlock changes are per-renderer, not per-material,
          so multiple enemies sharing the same material can flash
          independently without creating material instances.

====================================================
KEY DESIGN DECISIONS
====================================================

1. WHY WE PARENT THE WEAPON ON IMPACT
   When the weapon embeds in a surface, we call SetParent(collision.transform).
   This makes the weapon a child of whatever it hit. If that surface is a
   moving platform, rotating door, or swinging bridge, the weapon moves with
   it automatically. Without parenting, we would need to track the surface's
   movement every frame and manually update the weapon position — fragile,
   error-prone, and breaks on complex motion paths. Unity's transform
   hierarchy handles all of this for free.

2. STATE-BASED PHYSICS TRANSITIONS
   - Equipped: Kinematic ON (no forces, follows hand via parenting)
   - Thrown: Kinematic OFF (physics-driven flight via AddForce)
   - Embedded: Kinematic ON (frozen in place, parented to surface)
   - Recalling: Kinematic ON (script-driven Bezier interpolation)

   The key insight: always zero velocity BEFORE toggling isKinematic.
   If you toggle first, stored velocity creates a frame-1 impulse that
   can push the player or launch the weapon unpredictably.

3. VISUAL PIVOT SEPARATION
   Spin rotation is applied to a child "VisualPivot" object, not the weapon
   root. This ensures transform.forward always points in the movement
   direction, which is needed for: facing the throw direction, pointing
   toward the hand during recall, and orienting correctly when embedded.

4. IPHYSICSADAPTER USAGE
   All Rigidbody operations go through IPhysicsAdapter (AddForce, Velocity,
   SetKinematic). This follows the framework's physics-agnostic pattern.
   The weapon currently uses Physics3DAdapter, but switching to a future
   Physics2DAdapter would require zero code changes in the weapon itself.

5. BEZIER CURVE RETURN PATH
   A straight-line return is boring. A quadratic Bezier curve with an
   upward and sideways control point offset creates an organic arc that
   feels like the weapon is "cutting through the air" on its way back.
   The curve accelerates over time for the signature "snap" catch.

6. WEAPON SWITCHING (AXE / SPEAR)
   The demo supports multiple weapon variants (Axe and Spear) using the
   same BoomerangWeapon_UMFOSS component with different BoomerangConfig
   ScriptableObject assets. Switching is handled by BoomerangDemoController:
   - Only allowed when the current weapon is in the Equipped state
     (prevents switching while a weapon is mid-flight or embedded).
   - The current weapon's GameObject is deactivated, the next is activated.
   - Each weapon has its own physics, collider, visual mesh, and config.
   - The Spear config has: higher throw force (40 vs 30), longer range
     (35 vs 25), no spin (Vector3.zero rotation), more damage (35 vs 25),
     flatter return arc (arcHeightRatio 0.15 vs 0.3).
   - This demonstrates the ScriptableObject config pattern: zero code
     changes needed for entirely different weapon behaviors.

7. ROTATION PER SECOND (Vector3)
   RotationPerSecond is a Vector3, not a float, allowing per-axis control
   of the spin. The Axe uses (0, 0, 1080) for a forward-axis tumble.
   The Spear uses (0, 0, 0) for no spin — spears fly straight like a
   javelin. A hypothetical shuriken could use (0, 1080, 0) for a flat
   horizontal spin. WeaponVisualHandler.Spin() applies this as a single
   Rotate(degreesPerSecond * deltaTime, Space.Self) call.
